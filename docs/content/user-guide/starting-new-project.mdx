# Starting a New Agentic Project

This guide lays out a new agent project on top of Ark and tests it end to end. It assumes the [Quickstart](/quickstart) is done — Ark is running and `ark`, `kubectl` work against your cluster.

## Project Layout

Ark provides the runtime; your project — agents, tools, teams, queries — lives in its own directory (and usually its own repository). That separation lets each team evolve its agents independently of the platform.

There's no required layout. The convention below is a starting point.

## Prerequisites

- Ark installed
- `kubectl` configured
- A model named `default` (`ark models create default`)

```bash
ark --version
kubectl get pods -n ark-system
kubectl get models
```

## Step 1: Directory

```bash
# Create a new project
ark generate project my-agents
```

You can finalize your initial setup through one of the following ways:


### Option 1: Non-Interactive Mode

To skip prompts and use defaults:

```bash
ark generate project my-agents -d . --no-interactive --skip-models --skip-git
```

### Option 2: Interactive (Recommended for getting started)

```bash
The generator will ask you several questions to customize your project:

**1. Project Type:**
- `Project with samples (recommended for getting started)` - Includes example agents, teams, queries, and tools
- `Empty project (just the structure)` - Creates only the directory structure

**2. Namespace:**
- Kubernetes namespace for deployment (default: project name)

**3. Model Configuration (optional):**
- Choose which LLM providers to configure:
  - `Default OpenAI` - Standard OpenAI models
  - `AI Gateway` - Custom AI gateway endpoint
  - `All providers` - Configure multiple providers
  - `None` - Skip model configuration

**4. Git Setup (optional):**
- Initialize git repository
- Create initial commit
- Configure git user name and email
```

This creates a project structure with:

```
my-agents/
├── agents/          # Agent definitions
├── teams/           # Team configurations
├── queries/         # Query examples
├── models/          # Model configurations
├── tools/           # Tool definitions
├── mcp-servers/     # MCP server configurations
├── scripts/         # Utility scripts
├── templates/       # Component templates
├── tests/           # Test configurations
├── docs/            # Project documentation
├── Chart.yaml       # Helm chart definition
├── values.yaml      # Helm chart values
├── Makefile         # Build and deployment targets
├── .env             # Environment configuration
└── README.md        # Project README
```


Now you can find a default model configuration in the `models/` directory. Alternatively, you can copy a model configuration template from the `samples/models/` directory of the ARK repository.
Then, edit the `.env` file to add your API keys and other model related variables and make sure to source it.

## Deploying Your Project to ARK

After generating your project, deploy it to your ARK cluster:

### Using Make (Recommended)

```bash
# Deploy using Helm with custom resources
make install

# Or deploy to a specific namespace
make install NAMESPACE=my-team-agents

# Apply only custom resources (agents, teams, queries)
make resources-apply
```

### Using Helm Directly

```bash
# Install as a Helm chart
helm install my-agents . --namespace my-team-agents --create-namespace

# Upgrade an existing deployment
helm upgrade my-agents . --namespace my-team-agents
```

### Using kubectl

```bash
# Apply individual resources
kubectl apply -f agents/
kubectl apply -f teams/
kubectl apply -f queries/

# Or apply all YAML files
kubectl apply -f . -R
```

### Verify Deployment

```bash
# Check deployed resources
kubectl get agents,teams,queries -n my-team-agents

# Check agent status
kubectl get agents -n my-team-agents -o wide

# View agent logs
kubectl logs -l app=ark-api -n my-team-agents --tail=100
```

## Step 2: Generate Your First Agent

Now use the generator to create an agent:

```bash
# Generate an agent (interactive)
ark generate agent customer-support
```

### Interactive Prompts

The agent generator will ask:

**1. Agent Name:**
- Name for your agent (e.g., `customer-support`)

**2. Generate Test Query:**
- Option to create a sample query for testing

### Non-Interactive Mode

```bash
# Skip prompts
ark generate agent customer-support --no-interactive
```

This creates `agents/customer-support-agent.yaml` with dummy description and prompt, and the reference to default model. Edit the generated agent configuration:
Remember to set the `namespace` field to match the namespace of the project.

```yaml
mkdir -p my-agents/{agents,teams,queries,tools,mcp-servers}
cd my-agents
```

From here, every resource is a YAML file you write and apply with `kubectl`.

## Step 2: First Agent

```bash
cat > agents/customer-support.yaml <<'EOF'
apiVersion: ark.mckinsey.com/v1alpha1
kind: Agent
metadata:
  name: customer-support
spec:
  description: Handles customer inquiries and support tickets
  modelRef:
    name: default
  prompt: |
    You are a helpful customer support assistant.
    Help users with their questions and issues professionally.
EOF

kubectl apply -f agents/customer-support.yaml
kubectl get agent customer-support -o jsonpath='{.status.phase}'
# ready
```

## Step 3: Tool

An HTTP tool that calls the National Weather Service:

```bash
cat > tools/get-forecast.yaml <<'EOF'
apiVersion: ark.mckinsey.com/v1alpha1
kind: Tool
metadata:
  name: get-forecast
spec:
  type: http
  description: Get weather forecast for a location and time period
  inputSchema:
    type: object
    required: ["office", "gridX", "gridY"]
    properties:
      office:
        type: string
        description: "Weather office identifier (e.g., TOP)"
      gridX:
        type: integer
        description: "Grid X coordinate"
      gridY:
        type: integer
        description: "Grid Y coordinate"
  http:
    url: https://api.weather.gov/gridpoints/{office}/{gridX},{gridY}/forecast
EOF

kubectl apply -f tools/get-forecast.yaml
```

See [Tools](/user-guide/tools) for MCP, builtin, and agent-as-tool variants.

## Step 4: Team

Pick a strategy — each one expects a slightly different shape. See [Team Strategies](/user-guide/samples/teams/team-strategies) for the full reference.

### Sequential

Members run in declared order, one pass each:

```yaml
apiVersion: ark.mckinsey.com/v1alpha1
kind: Team
metadata:
  name: support-team-sequential
spec:
  description: Triage, then specialist
  strategy: sequential
  members:
    - type: agent
      name: customer-support
    - type: agent
      name: order-tracker
    - type: agent
      name: escalation-handler
```

### Sequential with Loops

Cycle through members until `maxTurns` is reached. `loops: true` and `maxTurns` must be set together:

```yaml
apiVersion: ark.mckinsey.com/v1alpha1
kind: Team
metadata:
  name: support-team-loops
spec:
  description: Each member contributes in turn until maxTurns
  strategy: sequential
  loops: true
  maxTurns: 5
  members:
    - type: agent
      name: customer-support
    - type: agent
      name: order-tracker
```

`strategy: round-robin` is deprecated and silently migrated to `sequential` with `loops: true`. Use the form above directly.

### Selector

A nominated agent picks the next member. `selector.agent` and `maxTurns` are required. Loops are not allowed on selector teams. The `selectorPrompt` should instruct the agent to use the built-in `select-next-speaker` tool:

```yaml
apiVersion: ark.mckinsey.com/v1alpha1
kind: Team
metadata:
  name: support-team-selector
spec:
  description: Selector agent picks the right specialist
  strategy: selector
  maxTurns: 10
  members:
    - type: agent
      name: customer-support
    - type: agent
      name: order-tracker
  selector:
    agent: planner-agent
    selectorPrompt: |
      Choose the team member best suited to respond next.

      Available roles:
      {{.Roles}}

      Conversation history:
      {{.History}}

      Use the select-next-speaker tool to make your selection.
```

`strategy: graph` is deprecated and migrated to plain `sequential` with edges discarded. For graph-style routing inside a selector team, see [Team Strategies](/user-guide/samples/teams/team-strategies).

Apply the chosen team:

```bash
kubectl apply -f teams/support-team-sequential.yaml
kubectl get team support-team-sequential
```

## Step 5: MCP Server (Optional)

Expose external capabilities via the Model Context Protocol:

```bash
cat > mcp-servers/example.yaml <<'EOF'
apiVersion: ark.mckinsey.com/v1alpha1
kind: MCPServer
metadata:
  name: example-mcp
spec:
  address:
    value: "http://example-mcp.default.svc.cluster.local:8080/mcp"
  transport: http
  timeout: 1m
EOF

kubectl apply -f mcp-servers/example.yaml
```

For authentication and OAuth, see [MCP Servers](/user-guide/samples/mcp-servers).

## Step 7: End-to-End Testing

### Bash smoke script

```bash
cat > test-workflow.sh <<'EOF'
#!/bin/bash
set -e

kubectl apply -f agents/
kubectl apply -f tools/
kubectl apply -f teams/

kubectl wait --for=jsonpath='{.status.phase}'=ready agent/customer-support --timeout=60s

kubectl apply -f queries/test-support.yaml
kubectl wait --for=jsonpath='{.status.phase}'=done query/test-support --timeout=120s

RESPONSE=$(kubectl get query test-support -o jsonpath='{.status.responses[0].content}')
[ -n "$RESPONSE" ] || { echo "FAIL: empty response"; exit 1; }

echo "PASS"
echo "Response: $RESPONSE"
kubectl delete query test-support
EOF

chmod +x test-workflow.sh
./test-workflow.sh
```

### Chainsaw (recommended)

Declarative, repeatable tests via [Chainsaw](https://kyverno.github.io/chainsaw/):

```yaml
apiVersion: chainsaw.kyverno.io/v1alpha1
kind: Test
metadata:
  name: customer-support-workflow
spec:
  steps:
    - name: apply-resources
      try:
        - apply:
            file: agents/customer-support.yaml
        - apply:
            file: queries/test-support.yaml

    - name: wait-for-agent-ready
      try:
        - assert:
            resource:
              apiVersion: ark.mckinsey.com/v1alpha1
              kind: Agent
              metadata:
                name: customer-support
              status:
                phase: ready

    - name: verify-query-done
      try:
        - assert:
            resource:
              apiVersion: ark.mckinsey.com/v1alpha1
              kind: Query
              metadata:
                name: test-support
              status:
                phase: done
                (length(responses) > `0`): true
                (length(responses[0].content) > `10`): true

    - name: cleanup
      try:
        - delete:
            ref:
              apiVersion: ark.mckinsey.com/v1alpha1
              kind: Query
              name: test-support
```

Run it:

```bash
chainsaw test --test-file chainsaw-test.yaml
```

## Step 8: Iterate

Add agents as YAML files and apply the directory:

```bash
# agents/order-tracker.yaml, agents/product-recommender.yaml, ...
kubectl apply -f agents/
```

Wire them into a richer team:

```yaml
apiVersion: ark.mckinsey.com/v1alpha1
kind: Team
metadata:
  name: full-support-team
spec:
  description: Complete customer support workflow
  strategy: sequential
  members:
    - type: agent
      name: customer-support
    - type: agent
      name: order-tracker
    - type: agent
      name: escalation-handler
```

## Best Practices

### Organise by feature

```
my-agents/
├── features/
│   ├── customer-support/
│   │   ├── agents/
│   │   ├── tools/
│   │   ├── queries/
│   │   └── tests/
│   └── order-processing/
│       ├── agents/
│       ├── tools/
│       └── tests/
```

### Version control

```bash
git init
git add .
git commit -m "feat: initial Ark project setup"
```

### Per-environment overlays with Kustomize

```yaml
apiVersion: kustomize.config.k8s.io/v1beta1
kind: Kustomization
resources:
  - agents/
  - tools/
  - teams/
namespace: my-agents-prod
```

### Continuous testing in CI

```yaml
name: Test Ark Agents
on: [push, pull_request]
jobs:
  test:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - name: Setup K3s
        run: curl -sfL https://get.k3s.io | sh -
      - name: Install Ark
        run: ark install -y
      - name: Run Tests
        run: chainsaw test --test-dir tests/
```

## Troubleshooting

### Agent stuck before `ready`

```bash
kubectl describe agent customer-support
kubectl logs -n ark-system -l app=ark-controller
kubectl get models
```

### Query stuck

```bash
kubectl get query test-support -o yaml
kubectl get query test-support -o jsonpath='{.status.error}'
kubectl get events --field-selector involvedObject.name=test-support
```

### Tool returns nothing

```bash
kubectl get tool get-forecast
kubectl describe tool get-forecast
```
